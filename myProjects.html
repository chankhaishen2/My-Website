Sentiment Classification of Patient Medication Feedback Text using Sentiment Lexicon, Bag of Words anf Term Frequency-Inverse Document Frequency

Category: Artificial intelligence, Natural language processing

Language: Python

Software: Jupyter Notebook

Date: January 2026

Institution: School of Computer Sciences, Universiti Sains Malaysia

Keywords: Machine Learning, Bag of Words (BOW), Term Frequency-Inverse Document Frequency (TF-IDF), Valence Aware Dictionary for Sentiment Reasoning (VADER), Text Preprocessing, Regular Expression Cleaning, Tokenization, Part of Speech Tagging, Lemmatisation, Stop Word Removal, Scikit Learn, Multinomial Naive Bayes, Logistic Regression, Multi-layer Perceptron, Random Forest, Model Training, Model Fine Tuning

This project is the project for Text and Speech Analytics subject from October 2025 to January 2026. This is an individual project. The purpose of this project is to investigate whether adding sentiment lexicon feature into the classical Bag of Words (BOW) or Term Frequency-Inverse Document Frequency (TF-IDF) will improve the performance of sentiment classification of patient medication feedback text. The dataset used in this project originates from https://www.kaggle.com/datasets/mukeshdevrath007/drugs-and-conditions-patient-voices-2-8l. The patient medication feedback text from the content attribute in the dataset is preprocessed. The preprocessing process involves regular expression cleaning, lowercasing, tokenzation, part of speech tagging, lemmatisation and stop word removal. The 10-scale rating in the rating column in the dataset is converted to 2-scale sentiment (1-4: negative, 5-10: positive). A baseline using VADER sentiment lexicon only (no machine learning) is set for this project and the macro average F1 is 0.5884. Macro average F1 is used to evaluate the performance of the models because the class distribution is imbalanced. 4 feature representations, namely BOW, BOW + VADER, TFIDF and TF-IDF + VADER, are experimented. 3 classical machine learning algorithms, namely logistic regression, multinomial naive Bayes and multi-layer perceptron, are tested. Random forest is also used initially, but in the first experiment with BOW feature representation, random forest has lower performance than the baseline (0.4075), so it is dropped in the subsequent experiments. It is found that all machine learning models (except random forest) outperformed the baseline. This shows machine learning is an effective method for sentiment classification of patient medication feedback. The best performing model is multi-layer perceptron with BOW + VADER feature representation, with test macro average F1 of 0.8300. For logistic regression, TF-IDF feature representation outperforms BOW feature representation because logistic regression benefits from lower magnitude of the normalized values of TF-IDF. For multinomial naive Bayes and multi-layer perceptron, BOW feature resentation outperforms TF-IDF feature representation because multinomial naive Bayes is designed for raw count and multi-layer perceptron might make use of raw frequency to extract the hidden pattern. For 3 out of 6 cases, adding sentiment lexicon improves the performance because sentiment lexicon gives extra information which is useful for the model, whereas for another 3 out of 6 cases, adding sentiment lexicon degrades the performance because sentiment lexicon adds noise to the feature. So, the conclusion is adding sentiment lexicon information to BOW and TF-IDF does not always improve the performance, though sometimes it does. I would like to thank my lecturer, Assoc. Prof. Gan Keng Hoon for her guidance.

This project is Assignment 2 for Text and Speech Analytics subject from October 2025 to January 2026. This is an individual project. In this project, speech emotion recognition using Mel frequency cepstral coefficient (MFCC) and classical machine learning is explored. The dataset used in this project originates from https://www.kaggle.com/datasets/ahmedeabozaid/audio-sentiment-analysis. The short speech audio files in the dataset is converted to MFCC. There are six emotion labels in the dataset, namely anger emotion, fear emotion, disgust emotion, happy emotion, neutral emotion and sad emotion. Since the distribution of classes is balanced, accuracy is used to evaluate the models. The experiment is split into two rounds. The first round focuses on finding good machine learning model for speech emotion recognition based on mean of MFCC. The second round focuses on investigating the difference of the performance of the top three models in the first round when using mean + standard deviation of MFCC and mean of MFCC. In the first round, 11 machine learning algorithms, namely decision tree, support vector machine with linear kernel, support vector machine with polynomial kernel, support vector machine with radial basis function kernel, support vector machine with sigmoid kernel, k nearest neighbours, Gaussian naive Bayes, Bernoulli naive Bayes, logistic regression, multi-layer perceptron and random forest, are tuned and compared. The three best performing models are k nearest neighbours, with test accuracy of 0.5411, support vector machine with polynomial kernel, with test accuracy of 0.5269 and multi-layer perceptron, with test accuracy of 0.5000. In the second round, these three algorithms are used to train speech emotion recognition model based on mean and standard deviation of MFCC. The test accuracy of k nearest neighbours, support vector machine with polynomial kernel and multi-layer perceptron are 0.5813, 0.6144 and 0.5605 respectively. So, the performance is better when using mean and standard deviation of MFCC compared using mean of MFCC only. This might be because standard deviation of MFCC provides extra information on the speech acoustic features. The universal best performing model in this project is the support vector machine with polynomial kernel model. I would like to thank my lecturer, Dr. Nur Hana Samsudin for her guidance.

This project is the coursework for two subjects, namely Web Application Programming Interface (API) Development (Task 1, April 2023 to July 2023) and Security (Task 2, August 2023 to November 2023). In Task 1, a website that uses a third-party API, SportsDB, as its source of data is developed. This website allows users to view the information, such as time, location and results, of any soccer event worldwide. Then, the user can save the event in bookmarks for future reference. The user can access his own bookmarks and view the list of bookmarked events, update the result of any event and remove any event from bookmark. On the other hand, the adminstrator can view all normal users and remove any normal user. In Task 2, the security aspects of the website are reviewed. The website already implements attribute-based (login is required to access user-specific bookmark) and role-based (login is required to access administrator functionality) access control. The website also already has password hashing by using Message Digest algorithm 5 (MD5). After analysing the existing website and researching the available security options, improvements are made in two aspects. Firstly, the password is hashed using Password-Based Key Derivation Function 2 (PBKDF2) before stored in the database. The purpose of password hashing is to prevent the password from being revealed in case of the database is leaked. Before being hashed, the password is also appended with a random string called salt. Adding different salt to password of different user makes rainbow table attack (pre-compute the hashed passwords) less efficient. After salting, brute force attack is still efficient due to high computing power processor. So, the password hashing is made into a computationally heavy process by applying key stretching. So, PBKDF2 is used. Secondly, password rules, namely "must have numbers", "must have upper-case and lower-cse characters", "must have special characters", and "must have at least 10 characters" are implemented and a simple password strength indicator is developed. The first three rules are implemented to overcome dictionary attack (guessing the password from dictionary words) by introducing randomness in the password. The last rule (password length) is implemented to make brute force attack harder by making the password longer. When the password is longer, the number of all possible passwords is larger. As a reminder to the user, a password strength indicator based on the compliance of each rule and the password strength is developed on the registration page. The password strength is calculated based on password length and number of possible characters. I would like to thank my lecturers, Mr. Shahriman Mohd Said (Web API Development) and Mr. R.K. Three of the six questions are related to parallel programming whereas three other questions are related to distributed programming. For parallel programming, part A is about printing "Hello World" from different threads and specifying the number of threads through command line and source code; part B is about comparing static scheduling and dynamic scheduling in a parallel for loop executing vector addition based on the time used (the result is static scheduling is faster than dynamic scheduling because the workload of vector addition is almost equal in each iteration); part C is about comparing the time used for parallel for loop with different number of threads (and also serial) and parallelizing different parts (outetmost loop and innermost loop) in matrix multiplication (the result is outermost loop parallelization is faster than innermost loop parallelization because the number of iteration in innermost loop parallelization is thousand times more than that of outermost loop parallelization; another observation is when size of matrix is larger, parallelization is more efficient). For distributed programming, part A is about specializing the number of processes and printing "Hello" from different processes; the first part of part B is about each slave process sends a message to the master process and the master process prints the messages, whereas the second part is similar to the first part but this time each slave process sends a different message; the first part of part C is the master process sends a message to each slave process and each slave process prints the message, whereas the second part is about the explaining the usage of message tags (the master process sends a message with tag "100" but the slave process waits for message with tag "101" so the program hangs, and the task is explains why the program hangs). The report was slightly edited on March 2025 to include additional explanation on the observation of time used for matrix multiplication using different matrix sizes, number of threads and innermost loop parallelization and outermost loop parallelization.
